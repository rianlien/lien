# 色グラデーションアプリ — 内部設計（招待・非同期回答フロー）

実装本体:
- `app/mesh-gradient/lib/participants.js` — セッション/参加者に対する業務ロジック（作成・追加・確定・削除・
  ロック判定）。永続化方式から独立した純粋関数として書いてあり、ローカル開発とVercelデプロイの両方で共有する。
- `app/mesh-gradient/lib/file-store.js` — ローカル開発用の永続化（JSONファイル、セッションごとに1ファイル）。
- `app/mesh-gradient/lib/kv-store.js` — Vercelデプロイ用の永続化（Upstash Redis）。
- `app/mesh-gradient/server.js` — ローカル開発用サーバー（Node httpモジュール、file-storeを使う）。
- `app/mesh-gradient/api/**` — Vercelデプロイ用のserverless functions（kv-storeを使う）。
- `app/mesh-gradient/public/index.html`（発起人）、`public/reply.html`（回答者）、`public/mesh-render.js`
  （描画エンジン、発起人・回答者の両ページで共有）。

各処理のWHYはコードコメントに書いてあるため重複させない。ここに書くのは、要件をどの技術選択で満たすことにしたか、
という設計判断のみ。

## 永続化方式の変遷: 自前サーバー＋JSONファイル → ローカル/Vercelで共有ロジック＋Upstash Redis

当初（発起人自身の検証段階）は、「友達の返信が発起人側で自動反映されてほしい」という要件（往復リンク方式では
満たせない）から状態をサーバー側に持つ構成が必須と判断し、DBミドルウェア（SQLite等）や外部BaaS
（Firebase/Supabase等）を選ばず、自前サーバー＋JSONファイルへの素朴な読み書きにした
（fable-advisorの助言:「実際に配布する段階の判断と合わせて後から決めた方がいい」）。

その後、「実際に友達に招待リンクを送って回答してもらう」段階に進むことになり、ホスティング判断
（fable-advisorへ再相談）の結果、GitHub Pages（静的ファイルのみ配信）単体ではバックエンドが動かせないため、
フロント・バックエンドをまとめて動かせるVercelを使うことにした。VercelはNode.jsのサーバーではなく
serverless functionsとしてAPIを実行するため、ファイルシステムへの永続化は使えない
（サーバーレス環境は実行のたびに別インスタンスで動きうるため、ローカルディスクへの書き込みが確実に残らない）。
そこで:

- 業務ロジック（`lib/participants.js`）を、ストレージの実装から完全に独立した純粋関数として切り出した
  （引数で渡された`session`オブジェクトを読み書きするだけで、ファイルかKVかを一切知らない）。これにより、
  ローカル開発（`server.js`＋`file-store.js`）とVercel本番（`api/**`＋`kv-store.js`）が**同じロジックを
  共有**し、ロック条件（respondedAtによる色/影響範囲/位置のロック等）が二重実装でズレる心配がなくなった。
- 永続化の単位を「セッション全体を1つの巨大なJSON（旧`data/sessions.json`）」から「セッションごとに1キー
  （ファイルなら1ファイル、KVなら1キー）」に変更した。KVはフラットなキーバリューストアなので、この粒度の
  方が自然に対応する。
- Vercel向けのKVには`@vercel/kv`ではなく`@upstash/redis`を使う。`@vercel/kv`は2024年12月にUpstash Redisへ
  統合され非推奨になっており、後継である`@upstash/redis`（`Redis.fromEnv()`）を直接使うのが現在の推奨。
  Vercelダッシュボードで「Upstash」インテグレーションをプロジェクトに接続すると、`Redis.fromEnv()`が読む
  環境変数（`KV_REST_API_URL`/`KV_REST_API_TOKEN`）が自動設定される。

## 回答者ページには「自分の分＋発起人の色1点だけ」しか渡さない

`GET /api/sessions/:sid/participants/:pid` は、その1参加者のレコードに加えて、同じセッションの発起人
（`isSelf: true`の参加者）の色だけを `initiatorColor` として追加で返す（`serializeForRespondent`関数）。
発起人の位置・他の友達の情報は一切含めない。回答者ページ（reply.html）はこのエンドポイントしか呼ばない。
発起人だけが使う `GET /api/sessions/:sid`（全参加者を返す）とはエンドポイントを分離しており、実装上
「回答者に発起人の色以外の情報を見せない」を、フロントエンドの表示を絞るだけでなく、サーバーが返すデータ
自体を最初から絞ることで担保している（フロント側の実装ミスで露出する経路を作らない）。

## ポストカードは「自分の色×発起人の色」の2点グラデーション

回答者に見せるポストカードは、単色ではなく、自分が選んだ色と発起人の色の2点を使い、既存のMesh Gradient
描画エンジン（`mesh-render.js`）にそのまま2点を渡して生成している（新しい描画ロジックを作らず、視覚表現の
一貫性を保つため）。配置は自分の色を上（y=0.28）、発起人の色を下（y=0.72）に固定し、reachを広め（0.42）に
とって中央でよく混ざるようにしている。位置に意味を持たせていない（発起人の実際の配置とは無関係）のは、
回答者に発起人の位置情報を一切渡さない、という制約と両立させるため。

## 発起人画面の自動反映: シンプルなポーリング

WebSocket等のプッシュ型通知ではなく、3秒間隔の `setInterval` による素朴なポーリングにした。利用規模が
発起人自身の検証のみで、リアルタイム性への要求も「自動で反映されてほしい」以上の具体的な遅延要件がないため、
実装・運用コストの低いポーリングで要件を満たせると判断した。ドラッグ操作中はポーリングの反映を止めている
（サーバーからの参加者一覧の丸ごと差し替えが、ドラッグ中の位置をローカルとサーバーとで奪い合う形になり、
操作がカクつく不具合が実装時に起きたため）。

## ロックの根拠は「色を設定したこと」ではなく「明示的に確定したこと」（`respondedAt` を確定操作でのみ立てる）

参加者レコードの `respondedAt` は、「色・影響範囲・位置は、明示的な確定操作をするまで自由に調整でき、
確定した後は一切変更できない」という要件のロック根拠として使う。サーバー側
（`PATCH /api/sessions/:sid/participants/:pid`）で、対象参加者に既に `respondedAt` が設定済みの場合、
`color`・`reach`・`x`・`y` のいずれの変更も拒否する（409を返す）。

当初は `color` を設定した**その場で** `respondedAt` を自動的にスタンプしていた。友達（回答者）にとっては
「送信する」ボタンを押す＝色を送る、が唯一の操作なので実質問題なかったが、発起人にとっては「色を選んだ後、
影響範囲を見ながら調整したり、色を選び直したりする」という自然な試行錯誤が起こりうる（明示要望）。
色を設定した瞬間にロックしてしまうと、この試行錯誤ができなくなる。

そこで `respondedAt` を立てるトリガーを、`color` の設定そのものから、**明示的な `confirm: true` フラグ**を
含むPATCHリクエストに切り離した。`color`・`reach`・`x`・`y` は、確定前であれば何度でも更新できる
（`confirm` を伴わないPATCHは値を更新するだけで `respondedAt` を立てない）。`confirm: true` が送られたときだけ、
その時点で `color` が設定されていることを条件に `respondedAt` をスタンプし、以後すべてのフィールドをロックする
（`color` が無い状態での確定は400エラーとして拒否——何も決めていないのに確定はできない）。

- **回答者（友達）**: reply.html の「送信する」ボタンは `{ color, confirm: true }` を1回のPATCHで送る。
  これまでどおり、ボタンを押した瞬間に色が決まり、同時にロックされる（体験は変わらない）。
- **発起人**: index.html には新たに「決定する」ボタンを用意した。色ピッカー・影響範囲スライダーはどちらも
  `confirm` を伴わない更新のみを送るため、ボタンを押すまでは何度でも調整できる。「決定する」ボタンだけが
  `{ confirm: true }` を送り、その時点の色・影響範囲・位置をまとめて確定させる。

フロントエンド側（index.html）: ロック済みかどうかの判定は、`p.color` の有無ではなく **`p.respondedAt` の有無**
で行う（自分は色が決まっていてもまだ確定していない状態がありうるため、`color`の有無だけでは判定を誤る）。
`respondedAt` が立っている参加者は、発起人自身も含めて①色スワッチを読み取り専用表示にする、②影響範囲
スライダーを「固定済み（値）」のテキスト表示に置き換える、③ドラッグのヒット判定から除外する。サーバー側の
拒否だけに頼らず、UI側でもその操作の導線自体を作らない（「エラーが出て初めて分かる」体験を避けるため）。
