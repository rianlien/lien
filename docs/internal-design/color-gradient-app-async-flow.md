# 色グラデーションアプリ — 内部設計（招待・非同期回答フロー）

<internal_design status="unrealized" name="async_invite_response_flow_design" requirements="async_invite_response_flow">

実装本体: `app/mesh-gradient/server.js`（API＋永続化）、`app/mesh-gradient/public/index.html`（発起人）、
`app/mesh-gradient/public/reply.html`（回答者）、`app/mesh-gradient/public/mesh-render.js`（描画エンジン、
発起人・回答者の両ページで共有）。各処理のWHYはコードコメントに書いてあるため重複させない。ここに書くのは、
要件をどの技術選択で満たすことにしたか、という設計判断のみ。

## 永続化方式: 自前サーバー＋JSONファイル（fable-advisorへの相談結果）

「友達の返信が発起人側で自動反映されてほしい」という要件（往復リンク方式では満たせない）から、状態をサーバー側に
持つ構成が必須と判断した。その上で、DBミドルウェア（SQLite等）や外部BaaS（Firebase/Supabase等）を今は選ばず、
`app/mesh-gradient/data/sessions.json` への素朴な読み書きにした理由:

- 現段階の利用規模が発起人自身による検証のみ（要件定義に明記）で、同時書き込みの衝突を心配する規模ではない。
- 外部サービスへの依存・アカウント契約は、実際に配布する段階の判断（ホスティング・公開範囲）と合わせて
  後から決めた方が良い、というfable-advisorの助言に沿った。
- データ契約（参加者のスキーマ: id/label/x/y/color/reach/isSelf/respondedAt）をストレージの実装から独立させて
  あるので、後でSQLite等に載せ替える場合もAPIハンドラの中身を差し替えるだけで済む設計にしている
  （`loadDb`/`saveDb`の2関数に永続化ロジックを閉じ込め、それ以外のコードはメモリ上のオブジェクトとして
  db を扱っている）。

## 回答者ページには「自分の分＋発起人の色1点だけ」しか渡さない

`GET /api/sessions/:sid/participants/:pid` は、その1参加者のレコードに加えて、同じセッションの発起人
（`isSelf: true`の参加者）の色だけを `initiatorColor` として追加で返す（`serializeForRespondent`関数）。
発起人の位置・他の友達の情報は一切含めない。回答者ページ（reply.html）はこのエンドポイントしか呼ばない。
発起人だけが使う `GET /api/sessions/:sid`（全参加者を返す）とはエンドポイントを分離しており、実装上
「回答者に発起人の色以外の情報を見せない」を、フロントエンドの表示を絞るだけでなく、サーバーが返すデータ
自体を最初から絞ることで担保している（フロント側の実装ミスで露出する経路を作らない）。

## ポストカードは「自分の色×発起人の色」の2点グラデーション

回答者に見せるポストカードは、単色ではなく、自分が選んだ色と発起人の色の2点を使い、既存のMesh Gradient
描画エンジン（`mesh-render.js`）にそのまま2点を渡して生成している（新しい描画ロジックを作らず、視覚表現の
一貫性を保つため）。配置は自分の色を上（y=0.28）、発起人の色を下（y=0.72）に固定し、reachを広め（0.42）に
とって中央でよく混ざるようにしている。位置に意味を持たせていない（発起人の実際の配置とは無関係）のは、
回答者に発起人の位置情報を一切渡さない、という制約と両立させるため。

## 発起人画面の自動反映: シンプルなポーリング

WebSocket等のプッシュ型通知ではなく、3秒間隔の `setInterval` による素朴なポーリングにした。利用規模が
発起人自身の検証のみで、リアルタイム性への要求も「自動で反映されてほしい」以上の具体的な遅延要件がないため、
実装・運用コストの低いポーリングで要件を満たせると判断した。ドラッグ操作中はポーリングの反映を止めている
（サーバーからの参加者一覧の丸ごと差し替えが、ドラッグ中の位置をローカルとサーバーとで奪い合う形になり、
操作がカクつく不具合が実装時に起きたため）。

## 色の送信・影響範囲（reach）・位置（x, y）の調整は、いずれも回答前までしかできない（`respondedAt` をロックの根拠に使う）

参加者レコードの `respondedAt` は、当初は表示上の状態管理のためだけに持たせていたが、「回答者はやり直せない」
「発起人は友達の色を見てから影響範囲・位置を調整できてはいけない」という要件に伴い、**アクセス制御の根拠としても
使う**設計に変更した。サーバー側（`PATCH /api/sessions/:sid/participants/:pid`）で、対象参加者が `isSelf: false`
（友達）かつ既に `respondedAt` が設定済みの場合、`color`・`reach`・`x`・`y` のいずれの変更も拒否する（409を返す）。
`isSelf: true`（発起人自身）にはこの制限をかけていない。

これら3項目（色・影響範囲・位置）を `color` と同じロック条件（`respondedAt`）にまとめたのは、いずれも同じ
バイアス問題（結果を見てから調整できてしまう）の一部だから。友達の影響範囲・位置は、回答前（色がまだ分からない間）
は自由に調整できる——発起人が「この人とは距離が近い／遠い」という関係性の感覚に基づいて決める分には、色の印象
によるバイアスが原理的に混入しないため。発起人が自分の色を選び直すこと・自分の影響範囲や位置を調整することは、
この制約の対象外（発起人は自分の色を最初から知っているため、色を見てからのバイアスという問題が原理的に発生しない）。

フロントエンド側（index.html）では、回答済みの友達をドラッグ対象の候補から外している（`pointerdown`のヒット
判定で `!p.isSelf && p.color` の点をスキップする）。サーバー側の拒否だけに頼らず、UI側でも操作の起点自体を
作らない、という設計方針は色・影響範囲のロックと同じ。

フロントエンド側: reply.html は `respondedAt` が既に設定されている参加者に対しては、色ピッカーや送信ボタン
自体を表示せず、送信済みのポストカードを読み取り専用で見せるだけにしている。index.html（発起人ページ）は、
回答済みの友達の行では影響範囲スライダーをテキスト表示に置き換え、操作できないようにしている。いずれも
サーバー側の拒否だけに頼らず、UI側でもその操作の導線自体を作らない（「エラーが出て初めて分かる」体験を避けるため）。
