# 色グラデーションアプリ — 要件定義（発起人による「完成」確定）

<requirements status="realized" name="session_completion" rfps="color_gradient_app">

## この要件定義がカバーする範囲

[async_invite_response_flow](./color-gradient-app-async-flow.md)（招待・非同期回答フロー）が実装済みになった後、
同要件定義の「未定義のまま残す項目」にあった「スナップショットとして『完成』を確定するタイミング」を確定する。

- 前提として async_invite_response_flow の発起人フロー・回答者フロー・「決定する」操作を理解している必要がある。

## 外部的な振る舞い・インタフェース

<human ts="2026-07-16T09:59:39+09:00">「完成」をスナップショットとして確定するタイミングは、友達をこれ以上追加するかどうか、みたいなところに完成ボタンを置いておきたい。</human>

### 発起人による「完成」確定
- 発起人は、友達をこれ以上追加しないと判断した時点で、セッション全体を確定する「完成」操作を行える。
- 「完成」は、発起人自身の色・影響範囲・位置を確定する既存の「決定する」操作（async_invite_response_flow参照）
  とは別レイヤーの操作である。「決定する」は参加者1件（発起人自身）の確定、「完成」はセッション全体の確定で
  あり、両者は独立に存在する。
- 「完成」した後は、新しい友達枠の追加・新規招待リンクの発行はできない。既存の招待リンクは、回答済みであれば
  引き続き読み取り専用のポストカードを表示する。
- 「完成」は不可逆の操作である（「決定する」「送信する」と同じく、明示操作による確定は以後変更できないという
  既存の方針に合わせる）。

<human ts="2026-07-19T11:00:00+09:00">（実装レビュー中に発見: 発起人が自分の色を一度も選ばずに「完成」できてしまい、完成版アートに発起人自身の
点が一切現れない状態になりうる。これは問題として修正する、という判断。）</human>

- **発起人自身の色が未設定のままでは「完成」できない**（色未設定の井戸は完成版アートの混色ロジック上
  存在しないものとして無視されるため、発起人自身の点が完成版から欠落してしまう）。「決定する」（respondedAt）
  までは要求しない——完成時点で色さえ選ばれていれば、その値で発起人自身の点も凍結される。

<human ts="2026-07-16T09:59:39+09:00">（AskUserQuestionでの回答）未回答の友達枠が残っている状態で「完成」させたくなったとき、完成と同時に自動で締め切る。</human>

- **未回答の友達枠が残っている状態で「完成」した場合、その枠は完成と同時に自動的に締め切る**（発起人が個別に
  削除してから完成する、という手順を要求しない）。
- 締め切られた枠の招待リンクを友達が後から開いたときに何を表示するか（エラーではなく「締め切られた」ことが
  伝わる文言等）は、[async_invite_response_flow](./color-gradient-app-async-flow.md)側のUXライティング課題に含める。

<human ts="2026-07-16T10:00:00+09:00">未回答枠の扱いについて：これは影響が表示されないだけ。そこにインクがたれなかっただけって感じにしたい。</human>

- 未回答のまま締め切られた友達枠は、完成版アート上では単に「その井戸に色が置かれなかった」状態として扱う。
  影響範囲・混色の計算上、存在しない井戸として扱われればよく、完成版アートを見せるための特別な除外処理を
  追加で作らない（[color-gradient-app.md](../internal-design/color-gradient-app.md)の混色ロジックが、
  色未設定の参加者をそのまま無視する形で自然に成立することを internal-design 側で確認する）。

### 回答済みの友達への「完成版」公開

<human ts="2026-07-16T10:00:00+09:00">回答済みの友達には、これまでの読み取り専用ポストカードと、完成版も見れるようにしたいかな。</human>

- 回答済みの友達が招待リンクを再訪すると、これまでどおりの「自分の色×発起人の色」の読み取り専用ポストカードに
  加えて、**セッション全体の完成版アート（発起人＋全友達の色が混ざった最終的なMesh Gradient）** も見られる。
  ただしこの露出は**「完成」した後にのみ**発生する。「完成」前に回答済みの友達が再訪した場合は、従来どおり
  自分のポストカードのみが見える（完成版はまだ存在しない）。

<human ts="2026-07-16T10:01:00+09:00">（AskUserQuestionでの回答）「完成」操作を押すこと自体を「回答済みの友達への公開同意」として定義していい？→完成=公開で統一。</human>

- **「完成」操作を行うこと自体が、回答済みの友達へ完成版アートを公開することへの同意にあたる**、と定義する。
  完成するかどうかとは別に「友達に見せるかどうか」を選べるトグルは設けない（完成＝公開で統一）。
  - Intent: [async_invite_response_flow](./color-gradient-app-async-flow.md)の「回答者には発起人の位置配置・
    他の友達の位置や色を一切見せない」という制約は、**回答フェーズ中**の制約として設けたものであり、理由は
    (a) 回答への影響防止（完成時点で全回答がロック済みのため、この理由は完成後は成立しない）と
    (b) 発起人の主観的な人間関係マップのセンシティブさ（この情報の持ち主は発起人自身であり、発起人が不可逆の
    「完成」を明示的に押す行為そのものが、その情報を友達へ公開する意思表示として十分と判断した）の2つ。
    完成後にこの制約の例外を設けても、Intentとの矛盾にはならない。
- **未回答のまま締め切られた友達枠には、完成版アートを公開しない**（回答済みの友達との非対称。招待に応じな
  かった／間に合わなかった枠に、応じた枠と同じ情報を渡す理由がないため）。回答済みかどうかで公開範囲が変わる
  ことの是非は、実際の利用で違和感が出た場合に見直す。

### 完成版アートのダウンロード

<human ts="2026-07-16T10:05:00+09:00">完成版アートのデータ形式は、humanがダウンロードできるものにしたい。</human>

- 完成版アートを見られる人（発起人・回答済みの友達）は、その完成版アートを画像ファイルとして手元に
  ダウンロードできる。
  - この前提は「完成版アートを閲覧できる対象＝ダウンロードできる対象」を同一にする、という単純化。閲覧は
    できるがダウンロードはできない、という中間状態は設けない。
- 画像フォーマット・解像度・ダウンロード操作のUI（ボタンの配置・ファイル名の付け方等）は internal-design に
  委ねるが、「ダウンロードして手元に残せる」こと自体は外部的な振る舞いとして確定する（内部設計・実装の都合で
  ブラウザ内表示のみに閉じてはならない）。

## 定義がなく、内部設計に委ねる項目

- 「完成」確定のデータ表現（セッションのどのフィールドで持つか、既存の`respondedAt`ロック機構との実装上の関係）は
  internal-design（[color-gradient-app-async-flow.md](../internal-design/color-gradient-app-async-flow.md)）に委ねる。
- 完成版アートのデータ形式は、上記「ダウンロードできる」という要件を満たす画像ファイル（例: PNG）として
  発起人側で描画し、セッションに保存する方式を internal-design で検討する。各参加者の色・位置・reachを
  JSONの生データとしてAPIから返す方式は採らない（「発起人の位置配置・他の友達の色を一切見せない」という
  回答フェーズ中の制約の理由＝センシティブさが完成後も参照される情報であるため、生のグラデーションパラメータ
  をそのまま露出しない）。画像の具体的な解像度・保存先（file-store/kv-storeどちらにどう置くか、既存の
  session1キー粒度とサイズの兼ね合い）は internal-design に委ねる。

## 未定義のまま残す項目（今回は確定しない）

- 締め切られた招待リンクを開いた友達への表示文言そのもの（UXライティング課題として
  [async_invite_response_flow](./color-gradient-app-async-flow.md)に集約）。
- reach（影響範囲）が小さい友達が、完成版アートから「自分の色が隅で小さい」ことを感じ取れてしまう点への
  配慮（文言・演出での緩和余地はあるが、今回は確定しない。UXライティング課題に合流）。
